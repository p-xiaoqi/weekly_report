# 周报自动生成系统 — 产品需求文档（PRD）

**版本**: v1.0  
**日期**: 2026-06-26  
**设计示例**: 以飞书生态为集成基座  
**技术约束**: 后端 Go 语言，无中间件，轻量级 Web 页面

---

## 1. 项目背景

### 1.1 场景
公司要求团队成员每周五提交周报，内容包括：本周完成工作、遇到的问题、下周计划。目前采用手工编写方式，通过邮件或文档工具提交。

### 1.2 痛点

| 序号 | 痛点 | 量化影响 |
|------|------|----------|
| 1 | **耗时严重** | 平均每人 30-60 分钟/周，20 人团队每周总耗时 15-20 小时 |
| 2 | **内容遗漏** | 需回忆本周工作，关键任务、代码提交、会议参与易遗漏 |
| 3 | **质量参差** | 有的详细有的简略，管理层阅读体验差，无法横向对比 |
| 4 | **与产出脱节** | 部分周报写得天花乱坠，但与实际 Git 提交、任务完成记录不符 |
| 5 | **数据孤岛** | 周报数据无法自动同步到项目管理系统，形成信息孤岛 |

### 1.3 目标用户
- **基层员工**：快速生成周报，减少重复劳动
- **团队负责人**：统一格式，掌握团队进度，催交提醒
- **管理层**：阅读体验一致，数据可量化，支持决策

---

## 2. 产品定位

**一句话定义**：基于飞书生态，自动汇聚 Git 代码、飞书任务、飞书日历会议记录，智能生成周报草稿的协同系统。

**核心价值主张**：
> "让数据自动说话，让人只补充思考"

**差异化亮点**：
- 深度集成飞书：身份、任务、日历、消息通知一站式打通
- 数据可验证：周报内容自动关联 Git 提交哈希、任务 ID、会议 ID
- 双向同步：周报数据可反向同步回飞书项目/任务系统

| 考核输入 | 系统功能映射 |
|---------|-------------|

| 任务完成记录 | 飞书任务/项目数据接入 |

| 周报模板 | 模板管理（按角色/团队定制） |

|---------|-------------|
| 自动生成的周报草稿（本周工作摘要+问题+计划） | 智能生成层（自动提取、去重、关联） |
| 支持人工编辑与批注 | 周报编辑页（富文本编辑器、批注、版本快照） |

| 历史周报查询与对比（本周vs上周） | 历史查询与对比分析（趋势图、数据对比） |
| 周报提交提醒（周五自动提醒未提交人员） | 提醒服务层（飞书机器人定时提醒） |

### 4.2 数据接入层（Data Ingestion）

**FR-1.1 Git 代码提交记录接入**
- 支持 GitLab / GitHub 平台，通过 API Token 授权访问
- 提取字段：仓库名、分支、提交哈希、提交消息、提交时间、作者、代码变更量（增删行数）
- 支持配置多仓库，按项目维度聚合展示
- 增量同步策略：首次全量，后续按时间增量拉取
- 数据校验：提交哈希唯一索引，防止重复导入

**FR-1.2 飞书任务完成记录接入**
- 通过飞书开放平台 API，读取用户本周在飞书项目/任务中的完成记录
- 提取字段：任务 ID、任务标题、任务描述、所属项目、完成时间、任务状态
- 支持接入飞书项目（Feishu Project）或飞书任务（Feishu Tasks）
- 权限要求：用户授权 `task:task:readonly` 权限

**FR-1.3 飞书日历会议记录接入**
- 通过飞书日历 API，读取用户本周的会议日程
- 提取字段：会议主题、会议时间、会议时长、参与人、会议类型
- 自动过滤：已取消会议不纳入；用户可标记"不纳入周报"的会议
- 支持读取会议纪要（如有权限），提取关键结论作为工作记录补充

**FR-1.4 手动补充记录**
- 提供 Web 表单，支持用户手动录入未被系统自动捕获的工作内容
- 字段：工作内容、所属项目、工作类型（开发/测试/文档/沟通/学习/其他）、耗时、备注
- 支持按日期录入（精确到某日），支持批量粘贴
- 手动记录与自动记录区分标识，便于后续统计自动覆盖率

**FR-1.5 数据同步管理**
- 支持手动触发同步（用户点击"刷新数据"）
- 支持定时自动同步（每日凌晨 2:00 自动同步前一天数据）
- 同步日志可视化：显示每次同步时间、数据源、成功/失败条数、失败原因
- 支持单条工作记录的编辑、删除、隐藏（不影响原始数据源）

### 4.3 智能生成层（Content Generation）

**FR-2.1 本周工作摘要自动生成**
- 输入：Git 提交、飞书任务、飞书日历、手动记录
- 处理逻辑：
  1. 按项目/仓库分组聚合
  2. 同一任务在 Git 和任务系统中重复出现时，以任务系统为准，Git 提交作为补充说明
  3. 同一项目的多次提交，合并为"完成 XX 功能，共提交 N 次代码"
  4. 生成结构化文本，包含：项目名称、任务/提交摘要、关联系统 ID（可点击跳转）
- 输出示例：
  ```
  【订单系统】
  - 完成任务 T-2024-0115-001：实现订单支付接口联调（飞书任务）
  - 提交代码 5 次：优化支付回调性能，修复超时 Bug（Git: order-service, commit: a1b2c3d）
  - 参与需求评审会议：确认支付对账逻辑（飞书日历：1 月 16 日 14:00-15:00）
  
  【用户中心】
  - 修复登录 Bug：验证码失效问题（T-002）
  ```

**FR-2.2 智能去重与关联**
- 自动识别重复内容：基于任务 ID、提交消息相似度、时间邻近性进行去重
- 自动关联：将 Git 提交与飞书任务通过提交消息中的任务 ID（如 `#T-123`）自动关联
- 用户可手动调整关联关系，系统学习用户偏好

**FR-2.3 问题填写智能引导**
- 系统无法自动识别问题，但提供智能引导：
  - 若本周有延期任务：提示"本周有 N 个任务延期，请说明原因"
  - 若本周有 Bug 修复记录：提示"本周修复了 N 个 Bug，是否遇到阻塞性问题？"
  - 若本周会议时长超过阈值：提示"本周会议时长较长，是否影响了开发进度？"
- 提供常见问题分类：技术难点、资源不足、需求变更、协作问题、外部依赖、其他
- 支持用户保存问题模板，便于复用

**FR-2.4 下周计划智能建议**
- 基于以下数据自动生成建议：
  - 本周进行中但未完成的任务（飞书任务状态"进行中"）
  - 已排期但未开始的任务（飞书任务状态"待处理"，开始时间在下周）
  - 本周遗留问题（用户填写的问题中标记为"待解决"）
- 输出示例：
  ```
  建议下周计划：
  - 继续完成【订单系统】任务 T-2024-0115-005：支付对账接口开发（进行中，已排期 60%）
  - 开始【用户中心】任务 T-2024-0115-010：登录鉴权优化（已排期，预计 1 月 22 日开始）
  - 跟进外部依赖：第三方回调延迟问题
  ```
- 用户可一键采纳建议，或手动增删改

**FR-2.5 人工批注与审阅**
- 在自动生成的内容旁添加批注框，支持记录"为什么修改"
- 高亮显示用户修改过的内容，显示修改人、修改时间
- 支持版本快照：保存每次编辑前的版本，支持回滚到任意历史版本
- 提交前支持预览：对比"自动生成版"与"最终提交版"的差异

### 4.4 模板管理层（Template Management）

**FR-3.1 模板基础管理**
- 支持创建、编辑、删除、启用/禁用模板
- 模板字段：名称、描述、适用范围、格式定义、创建人、创建时间
- 支持模板复制（基于现有模板创建新模板）
- 模板版本管理：修改模板时保留历史版本，可回滚

**FR-3.2 按角色定制模板**
- 系统预设角色模板：
  - **研发工程师**：本周工作（开发/测试/文档）、遇到的问题、下周计划、技术学习
  - **测试工程师**：本周工作（用例/测试/缺陷）、遇到的问题、下周计划、质量风险
   **产品经理**：本周工作（需求/调研/评审）、遇到的问题、下周计划、数据反馈
  - **设计师**：本周工作（设计/评审/走查）、遇到的问题、下周计划、设计沉淀
  - **团队负责人**：团队工作汇总、风险与问题、下周重点、资源需求
- 支持自定义角色，支持角色继承

**FR-3.3 按团队定制模板**
- 支持公司级默认模板、团队级模板、个人级模板三级体系
- 团队模板继承公司模板，个人模板可继承团队模板
- 团队负责人可设置本团队默认模板
- 支持模板强制使用或自由选择

**FR-3.4 模板字段配置**
- 可配置字段列表：本周工作、遇到的问题、下周计划、学习成长、团队贡献、项目风险、其他
- 字段属性：排序、必填/选填、字段类型（富文本/纯文本）
- 字段默认值：支持设置默认值，减少重复输入
- 富文本编辑器支持：Markdown、代码块、表格、链接、图片

**FR-3.5 模板变量系统**
- 内置变量：
  - `{{userName}}`：用户姓名
  - `{{weekRange}}`：本周日期范围
  - `{{teamName}}`：团队名称
  - `{{projectList}}`：本周涉及项目列表
  - `{{commitCount}}`：本周代码提交次数
  - `{{taskCount}}`：本周完成任务数
  - `{{meetingCount}}`：本周会议次数
- 变量在生成时自动替换，支持在模板中嵌入动态数据

### 4.5 输出与协同层（Output & Collaboration）

**FR-4.1 周报编辑与提交**
- 提供 Web 端富文本编辑器，支持 Markdown 语法
- 左侧显示"数据源面板"（Git 提交、任务、会议列表），右侧显示"周报编辑区"
- 支持从数据源面板拖拽内容到编辑区
- 支持自动保存草稿（每 30 秒自动保存，防止丢失）
- 提交前强制校验：必填字段不能为空
- 提交后锁定编辑（或进入"已提交"状态，需申请修改）

**FR-4.2 导出功能**
- 支持导出格式：
  - **Markdown（.md）**：纯文本格式，便于版本管理、Git 存档
  - **Word（.docx）**：保留模板样式，便于邮件发送、打印
  - **PDF（.pdf）**：固定版式，便于归档、不可篡改
- 导出时保留模板格式、字体、颜色、页眉页脚
- 支持批量导出：团队负责人可导出整个团队周报合并文档
- 导出文件命名规范：`周报_张三_2024W03_20240115.md`

**FR-4.3 历史周报查询与对比**
- 查询维度：
  - 按时间范围（本周/上周/本月/自定义）
  - 按人员（我/团队/全公司）
  - 按项目（筛选特定项目相关周报）
  - 按关键词搜索
- 本周 vs 上周对比：
  - 任务完成数对比（柱状图）
  - 代码提交量对比（折线图）
  - 会议参与次数对比
  - 问题数量趋势（折线图）
  - 工时投入分布（饼图）
- 支持生成"个人工作趋势报告"（月度/季度）
- 支持设置基准周，进行多周对比

**FR-4.4 飞书机器人提醒**
- 提醒策略：
  - 每周五 10:00：提醒未提交人员
  - 每周五 15:00：第二次提醒（未提交人员）
  - 每周五 17:00：最终提醒，抄送团队负责人
- 提醒内容：@人员 + 提醒文案 + 快捷链接（一键跳转到周报编辑页）
- 提醒渠道：飞书群机器人（支持多群配置）
- 提醒状态可视化：团队负责人可查看提交率统计、未提交人员名单
- 支持手动触发提醒（团队负责人可一键提醒指定人员）

**FR-4.5 双向数据同步**
- 正向：飞书任务/日历 → 周报系统（数据接入）
- 反向：周报系统 → 飞书任务/文档（数据回写）
  - 在周报中补充的工作记录，可一键创建为飞书任务
  - 周报中的问题记录，可自动创建为飞书项目的"风险/问题"事项
  - 已提交的周报，可自动归档到飞书云文档（按周/人/团队组织）
- 同步规则可配置：同步字段、同步目标、触发时机（提交时/手动）

**FR-4.6 权限与审批**
- 角色权限：
  - 管理员：系统配置、模板管理、全部数据查看、操作审计
  - 团队负责人：团队模板设置、团队成员数据查看、提醒发送、提交率统计
  - 普通成员：个人周报编辑、提交、导出、历史查看
- 审批流程（可选开启）：
  - 周报提交后需团队负责人审批
  - 审批通过后才算正式提交
  - 审批意见可批注在周报上

## 5. 非功能需求

### 5.1 技术架构要求
- **后端语言**：Go
- **数据库**：SQLite（本地零配置，适合快速启动；后续可平滑迁移至 PostgreSQL/MySQL）
- **缓存**：本地内存缓存（如 go-cache），满足单机场景
- **前端**：Go Templates + HTMX + Tailwind CSS（轻量级，无构建工具）
- **部署**：单二进制文件 + 配置文件，支持 Docker 一键部署

### 5.2 性能需求
- 周报生成响应时间 < 3 秒（Git + 飞书数据聚合）
- 页面加载时间 < 1 秒（缓存优化）
- 支持并发用户：50 人同时在线
- 数据同步：单次同步 < 30 秒，增量同步 < 5 秒
- 导出：Markdown 即时响应，Word/PDF < 5 秒

### 5.3 安全需求
- **认证**：飞书 OAuth 2.0 扫码登录（无密码体系，安全且便捷）
- **权限**：RBAC（基于角色的访问控制），数据隔离严格
- **传输**：全站 HTTPS（生产环境）
- **Token 安全**：Git Token、飞书 Token 加密存储（AES-256），不落地日志
- **审计日志**：记录关键操作（登录、提交、导出、同步、配置变更），保留 90 天
- **数据备份**：SQLite 文件自动备份（每日凌晨 3:00 复制备份文件）

### 5.4 可用性需求
- 系统可用性 > 99.5%（单机部署，无外部依赖时）
- 提供清晰的操作引导：首次登录向导、数据源配置引导、模板选择引导
- 提供帮助文档：FAQ、视频教程、快捷键说明
- 错误提示友好：网络异常、Token 过期、权限不足等场景给出明确提示和解决方案

### 5.5 扩展性需求
- **数据源插件化**：定义 `DataSource` 接口，新增数据源只需实现接口并注册（如后续接入 Jira、禅道、Tapd）
- **模板引擎可扩展**：支持自定义模板变量、自定义渲染函数
- **导出格式可扩展**：新增导出格式只需实现 `Exporter` 接口
- **通知渠道可扩展**：当前飞书机器人，后续可扩展邮件、企业微信、钉钉

## 6. 用户流程设计

### 6.1 首次使用流程（普通员工）

```
[开始]
  ↓
打开系统网址
  ↓
点击"飞书扫码登录" → 跳转飞书授权页
  ↓
手机飞书扫码确认 → 授权获取用户信息
  ↓
系统自动创建账号，进入首页
  ↓
首次登录向导：
  ├─ Step 1：配置 Git 数据源（粘贴 GitLab/GitHub Token，选择仓库）
  ├─ Step 2：配置飞书数据源（授权读取任务/日历，已登录态自动授权）
  ├─ Step 3：选择周报模板（根据角色推荐，可跳过使用默认）
```

## 7. 飞书集成详细方案

### 7.1 飞书 OAuth 登录

**流程**：
1. 用户访问系统，点击"飞书扫码登录"
2. 后端生成 `state`（随机字符串，防 CSRF），存入 Cookie
3. 后端拼接飞书授权 URL
4. 用户扫码确认授权
5. 飞书重定向至 `redirect_uri?code={CODE}&state={STATE}`
6. 后端校验 `state`，用 `code` 换取 `access_token`
7. 用 `access_token` 获取用户 `open_id`、`name`、`avatar`
8. 根据 `open_id` 查询本地用户，不存在则自动创建
9. 签发 JWT，设置 `HttpOnly` Cookie，跳转首页

**权限申请**：
- `contact:user.base:readonly`（获取用户基本信息）
- `task:task:readonly`（读取任务，可选）
- `calendar:calendar:readonly`（读取日历，可选）

### 7.2 飞书任务/日历数据接入

**任务数据**：
- 调用 `GET /open-apis/task/v1/tasks?user_id_type=open_id&completed_since={时间}`
- 提取字段：task_id、summary、description、completed_time、due_time

**日历数据**：
- 调用 `GET /open-apis/calendar/v4/calendars/{calendar_id}/events?start_time={}&end_time={}`
- 提取字段：event_id、summary、start_time、end_time、attendees

### 7.3 飞书群机器人提醒

**配置**：
- 在飞书群设置中添加"自定义机器人"
- 获取 `Webhook URL`
- 可选：开启签名校验，获取 `Secret`

**消息格式**：支持文本卡片和交互卡片，包含提醒文案和快捷链接按钮

**定时任务**：
- 使用 `robfig/cron/v3` 库，配置 `0 10 * * 5`（每周五 10:00）
- 扫描 `weekly_reports` 表，找出 `status != 'submitted'` 的用户
- 逐个调用飞书机器人 Webhook 发送提醒
- 记录发送日志到 `reminders` 表

### 7.4 双向同步（周报到飞书）

**场景 1**：周报提交后自动归档到飞书文档
- 调用 `POST /open-apis/doc/v2/create` 创建文档
- 将周报 Markdown 内容转换为飞书文档格式写入
- 存储文档 URL 到周报记录，便于后续查看

**场景 2**：手动记录一键创建飞书任务
- 用户在编辑周报手动记录时，可选择"创建为飞书任务"
- 调用 `POST /open-apis/task/v1/tasks` 创建任务
- 将任务 URL 回写到工作记录

## 8. 风险与应对

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 飞书 API 限流或变更 | 高 | 中 | 实现请求限流、本地缓存、异常降级；关注飞书开发者公告 |
| Git Token 泄露 | 高 | 低 | Token 加密存储（AES-256）、定期提醒更换、最小权限原则 |
| 用户数据源配置复杂 | 中 | 高 | 提供详细配置向导、一键测试连接、常见错误 FAQ |
| 自动生成内容质量低 | 中 | 中 | 提供人工编辑区、允许用户调整规则、持续优化聚合算法 |
| 单点故障（单机部署） | 中 | 中 | 每日自动备份 SQLite 文件；后续可迁移至主从架构 |
| 飞书应用审批未通过 | 高 | 低 | 提前与企业管理员沟通，确保应用权限申请符合公司安全规范 |

## 9. 附录

### 9.1 术语表

| 术语 | 说明 |
|------|------|
| 飞书 Open ID | 飞书用户在应用内的唯一标识 |
| 飞书 Union ID | 飞书用户在企业内的唯一标识，跨应用通用 |
| Webhook | 飞书机器人回调地址，用于向群/用户发送消息 |
| OAuth 2.0 | 开放授权协议，用于第三方应用安全获取用户信息 |
| JWT | JSON Web Token，无状态认证凭证 |
| 数据源 | 指 Git、飞书任务、飞书日历等外部系统 |
| 工作记录 | 从数据源同步的原始数据（提交、任务、会议） |
| 周报草稿 | 系统自动生成、待用户确认的周报 |
| 模板变量 | 模板中的动态占位符，如 `{{userName}}` |
| 双向同步 | 正向（外部系统 → 周报）与反向（周报 → 外部系统） |
